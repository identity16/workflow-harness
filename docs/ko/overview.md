# Workflow Harness

> 일하는 방식을 바꾸지 않고, AI의 힘을 활용하세요.

## 배경

모든 규모의 회사가 AI 워크플로우를 탐색하고 있지만, 대부분 같은 지점에서 멈춥니다: **기존 시스템 및 프로세스와의 통합**. "데모에서는 인상적인 AI"와 "우리 일상 업무에서 유용한 AI" 사이의 간극은 여전히 큽니다.

핵심 문제는 AI의 능력 부족이 아닙니다 — **마찰(Friction)**입니다. AI를 도입하려면 보통 새로운 도구를 배우고, 기존 프로세스를 바꾸고, 팀 전체를 설득해야 합니다. Linear, Asana, Jira, Figma, GitHub 위에서 이미 일하고 있는 개인에게 이건 현실적이지 않습니다.

**Workflow Harness**는 반대 방향에서 접근합니다: 기존 도구에 연결하여 AI로 강화하는 통합 인터페이스입니다. 같은 이슈, 같은 보드, 같은 데이터를 그대로 사용합니다 — 다만 뒤에서 AI가 워크플로우를 강화합니다. 팀원들은 기존 도구를 평소처럼 계속 씁니다. 그저 당신의 이슈가 더 잘 정리되어 있고, 컨텍스트가 풍부하고, 일을 더 빨리 해낸다는 걸 느낄 뿐입니다.

도입은 한 사람에게서 시작됩니다. 결과를 통해 퍼집니다.

## 철학

### 1. Zero Friction (제로 마찰)

도입 비용은 거의 0이어야 합니다. 한 사람이 팀의 동의, 프로세스 변경, 도구 마이그레이션 없이 바로 시작할 수 있어야 합니다. 기존 계정을 연결하면 바로 작업할 수 있습니다 — 같은 데이터, 더 나은 인터페이스. 팀원들은 아무것도 알 필요도, 바꿀 필요도 없습니다.

### 2. Observable (투명성)

AI는 블랙박스가 되어서는 안 됩니다. AI가 하는 모든 행동은 보이고, 추적 가능하고, 설명 가능해야 합니다. 사용자는 항상 AI가 *무엇을* 했는지, *왜* 했는지, *어떤 소스를* 사용했는지 확인할 수 있습니다.

### 3. Human-in-the-Loop (사람이 통제)

AI는 기본적으로 자율적으로 행동하지 않습니다. 사람이 AI가 무엇을 할 수 있는지, 언제 할 수 있는지, 그 결과물을 얼마나 신뢰할지를 통제합니다. 이 시스템은 AI의 자율성이 아니라 사람의 판단을 중심으로 설계됩니다.

### 4. Individual-first (개인 우선)

도입의 단위는 팀이나 조직이 아니라 한 사람입니다. 한 명의 개발자가 오늘 바로 시작할 수 있습니다. 팀원들이 결과를 보면 — 더 풍부한 이슈, 더 좋은 컨텍스트, 더 빠른 전달 — 그들도 원하게 됩니다. 팀 레벨 기능(공유 Trust Policy, 팀 대시보드)은 도입이 자라면서 자연스럽게 등장하지만, 전제조건이 되지는 않습니다.

## 핵심 개념

### Trust Level (신뢰 수준)

Workflow Harness가 생산하는 모든 산출물에는 **신뢰 수준**이 부여됩니다:

| 신뢰 수준 | 의미 | 예시 |
|---|---|---|
| `ai-generated` | AI가 사람의 검토 없이 생산 | 자동 수집된 컨텍스트, AI가 만든 이슈 초안 |
| `human-verified` | 사람이 검토하고 승인 | Q&A로 확인된 컨텍스트, 수동 승인된 이슈 |

이 구분은 단순한 기능 플래그가 아니라 **프로젝트 전체의 철학**입니다. 컨텍스트, 이슈, 제안, 코드 등 어떤 산출물이든 항상 신뢰 수준이 태깅되어, 그 정보를 사용하는 사람이 얼마나 신뢰할 수 있는지 판단할 수 있습니다.

### Trust Policy (신뢰 정책)

Trust Policy는 **AI가 자율적으로 행동할 수 있는 경우와 사람의 확인이 필요한 경우를 정의**합니다.

#### Adaptive Learning (적응형 학습)

Trust Policy를 수동으로 설정할 필요가 없습니다. 시스템이 **사용자의 행동에서 학습**합니다:

1. **시작**: 모든 것이 `human-required`로 시작. AI는 항상 확인을 요청.
2. **관찰**: 시스템이 Q&A 응답 패턴을 추적 — 어떤 유형의 이슈를 항상 승인하는지, 수정하는지, 거부하는지.
3. **제안**: 패턴이 감지되면 ("'chore' 라벨 이슈를 15번 연속 승인하셨네요"), 정책 변경을 제안.
4. **확인**: 제안을 승인하면 정책이 업데이트. 반복되는 Q&A 대신 한 번의 결정.

```
1주차:
  AI: "이 컨텍스트 맞나요?" → 나: "맞아"
  AI: "이 컨텍스트 맞나요?" → 나: "맞아"
  AI: "이 컨텍스트 맞나요?" → 나: "맞아"
  (모든 이슈에 대해 반복)

3주차:
  AI: "'chore' 이슈의 컨텍스트를 항상 승인하셨네요.
       앞으로 자동 보강할까요?"
  나: "그래"
  → 정책 업데이트: chore → ai-only

6주차:
  AI: "'docs' 이슈도 항상 승인하셨네요."
  나: "OK"
  → 정책 업데이트: docs → ai-only
```

이렇게 하면 **조직이 변화해도 시스템이 자동으로 적응**합니다. 라벨이 바뀌어도, 팀 컨벤션이 달라져도, 새로운 이슈 유형이 생겨도 — 학습 루프가 새로운 패턴을 자동으로 잡아냅니다. 유지보수할 설정 파일도, 업데이트할 매핑도 없습니다.

#### Manual Override (수동 오버라이드)

원하면 언제든 명시적 규칙을 설정할 수 있습니다:

```yaml
# 명시적 오버라이드 (선택사항)
rules:
  - match: { label: "security" }
    trust: human-required    # 항상, 무조건
```

#### 개인 → 팀 확장

처음에 Trust Policy는 **개인적**입니다 — 나의 행동에서 학습됩니다. 팀 도입이 늘어나면, **팀 레벨** 정책으로 확장하여 공유 기본값을 제공하면서 개인 오버라이드도 허용합니다.

```
개인 도입:
  개인 Trust Policy (내 행동에서 학습)
  └─ "나는 chore는 AI에게 맡기고, bug는 직접 확인"

팀 도입:
  팀 Trust Policy (팀 패턴에서 집계)
  └─ 개인 오버라이드 (멤버별)
```

### Unified UI (통합 UI / 프록시 아키텍처)

Workflow Harness는 새로운 작업 관리 도구가 아닙니다 — **이미 쓰고 있는 도구를 위한 더 나은 인터페이스**입니다. Linear, Asana, Jira에 API로 연결하여 다음을 제공합니다:

- 전체 이슈 관리 (생성, 수정, 정리) — 양방향 동기화
- AI 컨텍스트 보강 — 기존 도구를 쓰는 팀원에게는 보이지 않음
- 인라인 Q&A — 별도 대시보드가 아닌, 이슈 뷰 안에서 자연스럽게 검증
- Trust Level 배지 — AI가 한 것과 내가 확인한 것을 명확히 표시

```
┌─────────────────────────────────────┐
│  Workflow Harness UI                │
│  (내가 사용하는 인터페이스)             │
│                                     │
│  ┌──────────┐  ┌──────────┐        │
│  │ Kanban   │  │ Issue    │        │
│  │ Board    │  │ Detail   │        │
│  │          │  │ + Q&A    │        │
│  │          │  │ + Trust  │        │
│  │          │  │ + Context│        │
│  └──────────┘  └──────────┘        │
└──────────────┬──────────────────────┘
               │ API 동기화
┌──────────────┴──────────────────────┐
│  Linear / Asana / Jira              │
│  (데이터가 있는 곳)                    │
│  (팀원들이 계속 쓰는 곳)               │
└─────────────────────────────────────┘
```

### Context Engine (컨텍스트 엔진)

Context Engine은 **반자동 컨텍스트 보강**을 담당합니다. 이벤트가 발생하면(예: 이슈 생성), 엔진은:

1. **수집(Gather)** — 관련 코드, 과거 이슈, 문서 등 관련 정보를 모음
2. **확인(Check)** — 해당 이슈 유형의 Trust Policy를 확인
3. **분기(Branch)** — 신뢰 수준에 따라:
   - `ai-only` → 컨텍스트를 자동 첨부, `ai-generated`로 태깅
   - `human-required` → 인라인 Q&A를 통해 검증, `human-verified`로 태깅

핵심 설계 목표: **가능한 한 많은 컨텍스트를 자동으로 수집하되, 중요한 곳에서는 사람의 검증으로 품질을 통제한다.**

보강된 컨텍스트는 기존 도구로 동기화됩니다 — Linear에 있는 팀원들은 어떻게 된 건지 모르면서도 더 풍부한 이슈의 혜택을 받습니다.

### Origin-Aware Flow (출처 인식 흐름)

시스템은 산출물을 **누가 만들었는지**를 구분합니다:

```
사람이 만든 이슈:
  → Context Engine이 보강
  → Trust Policy에 따라 자동 or Q&A

AI가 만든 이슈:
  → Trust Policy 확인
  ├─ ai-only 정책 → 자동 활성화 (태그: ai-generated)
  └─ human-required 정책 → Draft → Q&A → 활성화 (태그: human-verified)
```

AI가 생성한 이슈는 워크플로우에 조용히 주입되지 않습니다. Trust Policy에 따라 사람의 승인이 필요할 수 있으며, 승인 전에는 기존 도구로 동기화되지 않습니다.

### Q&A Protocol (Q&A 프로토콜)

Q&A는 사람의 검증을 위한 메커니즘입니다. **최소한의 인지 부하**를 원칙으로 설계됩니다:

- 질문은 작고, 집중적이며, 빠르게 답할 수 있어야 함
- Q&A는 **이슈 뷰 안에서 인라인으로** 처리 — 컨텍스트 전환 없음, 별도 대시보드 없음
- 목표는 검증이 부담이 아니라 자연스럽게 느껴지도록 만드는 것

## 로드맵

### Phase 1: 코어 엔진 + Linear 통합

기반 시스템을 구축하고, 한 사람, 하나의 도구로 검증합니다.

**@workflow-harness/core**
- Context Engine — 다양한 소스에서 컨텍스트를 수집하고 보강
- Trust Policy — 규칙 기반 신뢰 수준 할당 (개인 범위)
- Q&A Protocol — 구조화된 인라인 검증 흐름
- Artifact Tagging — 모든 산출물에 신뢰 수준 메타데이터

**@workflow-harness/adapter-linear**
- Linear API 연동 (양방향 동기화)
- 이슈 ↔ 컨텍스트 매핑
- 투명한 동기화 — 팀원들은 Linear에서 보강된 이슈를 봄

**@workflow-harness/app**
- 통합 웹 UI — 칸반 보드 + 이슈 상세 + 인라인 Q&A
- Trust Level 배지 및 컨텍스트 뷰어
- 개인 Trust Policy 설정

### Phase 2: Task Tool 확장

Linear 패턴을 다른 작업 관리 도구로 복제합니다.

- **@workflow-harness/adapter-asana**
- **@workflow-harness/adapter-jira**
- 커뮤니티 기여를 위한 공유 어댑터 인터페이스/프로토콜
- 팀 레벨 Trust Policy 지원

### Phase 3: 작업 도구 통합

"컨텍스트 보강"에서 "실제 작업 지원"으로 확장합니다.

- **Figma 통합** — 디자인 컨텍스트가 개발 이슈로 흐름
- **GitHub 통합** — 코드 컨텍스트, PR 워크플로우
- **Claude Code Skills** — 보강된 컨텍스트를 기반으로 AI 워크플로우 자동화 트리거
